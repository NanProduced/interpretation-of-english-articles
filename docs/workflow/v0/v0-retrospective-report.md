# Workflow V0 架构回顾与问题诊断报告

## 1. V0 架构概述
Workflow V0 是一个基于 LangGraph 的线性串行系统，旨在通过「预处理 -> 路由 -> 核心标注 -> 翻译 -> 合并富化 -> 校验」的链路，将一段原始英文文本转化为包含词汇、语法、长难句和翻译的结构化教学数据。

### 1.1 核心链路
- **preprocess_v0**: 负责文本规范化、分句（Segmentation）及质量评估（Guardrails）。
- **core_agent_v0**: 单节点输出词汇、语法点和长难句的结构化 JSON。
- **translation_agent_v0**: 负责全文翻译、逐句翻译及关键短语翻译。
- **validate_node**: 对生成的标注进行引用校验（如检查 `sentence_id` 是否存在）。

## 2. 核心问题诊断 (Retrospective)
经过多轮联调与 Trace 分析，V0 版本在性能、稳定性和教学价值上均未达到预期。失败原因可收敛为以下五类：

### 2.1 结构定位错误 (Structural Misalignment)
- **坐标系混乱**：Span 偏移量计算完全依赖模型对字符串长度的感知，导致高亮错位频繁。
- **边界污染**：标注建立在不稳定的句子边界上。如果分句逻辑（Segmentation）与 LLM 的理解不一致（如 LLM 认为两句话应合并），会导致 `sentence_id` 引用失效。
- **过度设计**：`SentenceComponent` 要求解析到「主谓宾定状补」的绝对位置，超出了当前主流模型在复杂 JSON 生成任务中的稳定性极限。

### 2.2 教学目标偏离 (Pedagogical Misalignment)
- **「填表」压力**：Prompt 强迫模型在单次输出中填充大量字段（词汇、语法、难句、翻译等）。
- **价值稀释**：为了满足 Schema 的数量要求，模型倾向于抓取简单词汇或无意义的语法结构（如「直接引语」），而非提供真正有教学价值的讲解。
- **模型注意力竞争**：模型在努力满足「格式正确」时，自动降低了「教学质量」的权重。

### 2.3 Schema 贪婪过度 (Schema Greediness)
- **任务耦合**：`core_agent` 同时承担了筛选（选词）、分类（语法类型）、拆解（难句）和定位（Span）四重任务。
- **冗余逻辑**：`translation_agent` 与 `core_agent` 均需处理全文上下文，且在「关键短语翻译」与「词汇解释」上存在逻辑重叠，增加了整体输出负担。

### 2.4 Graph 链路过重 (Heavy & Sequential Graph)
- **高延迟设计**：全链路采用串行模式（`core -> translation -> merge`），无法发挥分布式生成的优势。
- **上下文冗余**：每个核心节点都在重复读取全文章节，导致 Token 消耗巨大，响应时间普遍在 80s-120s 之间。
- **状态膨胀**：`AnalyzeState` 随着节点执行不断累积中间产物，增加了调试和维护难度。

### 2.5 Fallback 策略错误 (Faulty Fallback Strategy)
- **内容污染**：`analysis/fallbacks.py` 在 LLM 失败时会生成「简化说明」、「待优化翻译」等伪数据。
- **副作用**：此类「为了保结构而填充的内容」对用户毫无价值，甚至会产生误导。宁可报错也不应输出糟糕的标注数据。

---

## 3. 技术指标复盘 (Evidence)
- **平均延迟**：80s - 150s (DeepSeek/Qwen 混合测试)。
- **失败模式**：`UnexpectedModelBehavior`（Schema 校验失败）和 `INVALID_SENTENCE_REFERENCE`（引用错位）。
- **Token 效率**：核心节点输出量常年维持在 3k-4k tokens，且其中大部分为冗余的结构字段。

---

## 4. 结论与 V1 导向
V0 的设计过于理想化，试图通过单次重型调用解决所有问题。
**V1 改进方向：**
1. **任务分治**：将「发现问题（提取）」与「解决问题（生成详情）」拆分为不同 Agent。
2. **并行加速**：解耦 Core 与 Translation，实现并行执行。
3. **锚点引用**：引入 `anchor_text` 匹配机制，摆脱绝对 Span 定位。
4. **高质量 Fallback**：宁缺毋滥，取消伪数据填充，引入分层降级。
