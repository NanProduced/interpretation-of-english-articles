# Workflow 摘要

来源文档：`AI 解读 Workflow 流程设计（草图）`

注意：该源文档仍未定稿。本文档只迁入当前已形成共识的部分，作为后端实现和验证的起点，而不是最终定版。

## 文档定位

本文档描述后端 AI 解读 Workflow 的业务骨架，目标是指导最小可运行版本的搭建与后续迭代。

当前原则是：

- 先做最小完成体
- 先验证输出质量和结构稳定性
- 再根据真实测试结果迭代 prompt、节点拆分和富化策略

## Workflow 的职责

Workflow 的职责是把：

- 英文文章文本
- 用户配置
- 请求元信息

转换为：

- 可解析
- 可渲染
- 可存储
- 可版本化

的结构化 JSON 输出。

它不负责：

- 前端渲染
- 账号体系
- 长期数据运营逻辑
- 词典型确定性知识的完整生成

## 当前核心设计原则

### 1. Schema 优先

输出结构必须先稳定，内容丰富度可以后补。前端和后端之间的第一约束是 Schema。

### 2. AI 只做语境相关任务

由 AI 负责：

- 语境义判断
- 语法结构分析
- 长难句拆解
- 翻译风格把控

尽量不用 AI 负责：

- 音标
- 基础释义
- 词表命中
- 规则映射

### 3. 少轮调用优先

在输出质量可接受的前提下，优先使用少量、可控、并行的调用，而不是过度拆分成大量 Agent。

### 4. 可校验、可重试、可降级

Workflow 必须具备：

- 输出校验
- 有上限的重试
- 明确的失败路径
- 必要时的降级结果

## 当前流程骨架

当前共识流程：

1. 合规审核
2. 文本预处理
3. Router 按 profile 和文本质量分流
4. 进入核心分析 Agent
5. 合并结果
6. 规则富化
7. 输出校验
8. 返回结构化 JSON

## 当前推荐的 Agent 切分

### core_agent

负责：

- 词汇标注
- 语法分析
- 长难句识别与拆解

这三者耦合较强，当前建议放在同一个核心 Agent 中。

### translation_agent

负责：

- 逐句翻译
- 全文翻译
- 关键短语翻译

翻译与标注耦合相对较弱，适合独立并行执行。

### discourse_agent

负责：

- 段落大意
- 文章结构
- 逻辑连接词
- 主题句

这是高阶能力，当前不作为最小版本的强依赖。

## Profile 策略

当前共识不是“按水平裁剪输出内容”，而是：

- 先做尽可能完整的标注
- 再根据用户 profile 做解释风格和展示优先级映射

这意味着不同用户可以共享一份基础分析结果，只在后处理阶段做优先级差异化。

## 预处理原则

V1 只允许做不会改变原文语义的清理：

- 去除 HTML 标签
- 空白归一化
- Unicode 规范化

V1 不应直接改写用户原文，也不应默默修正语法错误。

如果文本质量较差，应优先：

- 标记问题
- 触发降级
- 向用户解释风险

## 当前最值得警惕的工程问题

### 1. span 对齐

如果标注和原文位置无法稳定对齐，前端渲染会直接失真。

### 2. 输出结构稳定性

即使内容“看起来不错”，只要结构不稳定，就无法支撑可维护的结果页。

### 3. 长文延迟

小程序用户对等待很敏感，Workflow 需要从一开始就考虑并行、降级和按需加载。

## 当前未定项

以下内容仍需要在最小 Workflow 跑通后通过测试继续收敛：

- 最终输出 Schema 细节
- 校验失败后的重试策略
- 降级策略
- 各节点的延迟预算
- token 预算
- 缓存策略
- 模型分层与 fallback 策略

## 当前实施建议

当前后端实现应遵循下面的顺序：

1. 先定义最小输出 Schema
2. 先跑通 `core_agent + translation_agent`
3. 先建立最小校验与健康测试
4. 再决定是否需要新增 Agent、RAG 和更复杂的富化链路
