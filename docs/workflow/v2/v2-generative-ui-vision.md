# V2 Workflow 方案设想：从“数据填表”到“生成式教学 UI”

## 1. 背景与核心痛点 (Why V2?)

在 V1 版本的开发中，我们发现以下瓶颈：
*   **语义分类僵化**：LLM 强制将知识点归类为“词汇、语法、长难句”。这种分类虽逻辑清晰，但导致教学手段单一（均为“点击查词”或“底部卡片”），无法根据文章内容灵活调整解释方式。
*   **标注质量失真**：为了填满规定的 Schema 数组，LLM 往往会寻找基础词汇或语法进行“凑数”，导致中级读者的获得感较低。
*   **前后端联调滞后**：前端样式高度依赖后端的业务逻辑分类，导致前端开发必须等待后端 Schema 稳定后才能开始，缺乏表现力的探索。

## 2. V2 核心设想 (The Vision)

**核心转变：让大模型从「表单填写者」变为「教学编排者」。**

我们将不再定义业务意义上的数据结构（如 `vocabulary_annotations`），而是定义一套**「教学 UI 组件协议」**。

### 2.1 教学工具箱 (UI Component Box)
前端先行定义并开发一系列标准化的标注组件：
*   **InlineMark (内联标记)**：用于轻量级的词汇、短语高亮，点击唤起气泡。
*   **AnalysisCard (深度解析卡)**：插入在段落间的卡片，可展示树状结构图、对比列表等。
*   **ComparativeBridge (翻译对照桥)**：专门用于展示直译与意译的语感差异。
*   **ThoughtNote (批注随笔)**：模拟老师在书页边角的“碎碎念”。

### 2.2 工作流逻辑
1.  **前端**：维护组件库，并提供每个组件所需的 `props` 规范。
2.  **LLM**：根据文章难度，自主决定在哪个位置调用哪个组件。它返回的是一个**组件流 (Component Stream)**，其中包含了组件类型及其渲染所需的参数。
3.  **解析层**：前端收到组件序列，直接映射到对应的 UI 样式。

## 3. 预期价值

*   **体验升级**：界面呈现更具“AI 驱动”感，像是一位名师手绘的批注笔记，而非冷冰冰的查词工具。
*   **内容提质**：AI 将重心放在“如何讲好这个知识点”上，根据语境选择最合适的视觉表达。
*   **解耦开发**：前端可以独立开发更精美的组件并进行 Mock 调试，后端只需保证下发的 `props` 符合 UI 规范。

## 4. 关键讨论议题 (Open Questions)

为了落地该方案，我们需要共同讨论并解决以下核心问题：

### Q1：锚点定位与 ID 绑定 (Anchor Management)
*   **问题**：当 AI 返回一个 `InlineMark` 组件时，如何确保它高亮的文字在正文中精确匹配？
*   **讨论点**：是继续沿用 V1 的 `sentence_id` 范围？还是允许 AI 自由返回 `anchor_text`？如何处理 AI 输出微小误差导致的匹配失败？

**当前结论：**
*   V2 仍然保留 `sentence_id` 作为锚点定位的硬边界，不接受脱离句子范围的自由定位。
*   词级/短语级组件使用 `sentence_id + anchor_text + occurrence` 作为基础定位方式；句级组件只绑定 `sentence_id`；段间组件使用 `after_sentence_id` 或 `paragraph_id`。
*   不再把单个 `start-end` 绝对偏移作为 LLM 的输出目标。LLM 只负责返回原文片段，定位解析由本地完成。
*   针对 `so ... that`、`not only ... but also` 这类分隔式短语或不连续语法模式，必须支持 **span 列表 / 锚点列表**，不能再假设所有标注都能收敛成一个连续 span。
*   这也是 V1 测试暴露出的真实问题：LLM 在固定短语组合标注时会输出带 `...` 的数据，单一 span 匹配会直接失败。因此 V2 的锚点协议必须允许多段原文片段共同组成一个标注。

**下一步约束：**
*   前端实验阶段可先基于 `sentence_id + 子串匹配` 做验证。
*   正式联调时应由后端先做锚点解析和校验，前端尽量消费稳定的定位结果，而不是让前端独自承担全部定位容错。

### Q2：视觉冲突与密度控制 (UI Conflict)
*   **问题**：如果 AI 在同一行文字上既想放个气泡，又想放个下划线，视觉上如何排雷？
*   **讨论点**：是否需要一套“标注排版引擎”来自动分配空间？还是在 Prompt 中设定强硬的互斥规则？

**当前结论：**
*   视觉冲突不能只靠 Prompt 规避，必须存在一层本地“冲突消解 / 排版规则”。
*   同一锚点区域只允许一个主交互元素；同一段文本默认只允许一种 L0 视觉样式，不叠加多种强样式。
*   超过密度阈值时应自动降级，例如优先保留词级高亮，弱化句级说明，限制段间卡片数量。
*   组件可以由 AI 自由选择，但最终能否同时落位、是否需要降级，应由本地规则层决定。

### Q3：Prompt 的工程化与稳定性 (Prompt Reliability)
*   **问题**：如何防止 AI “自创”组件或漏掉必填的 `props`？
*   **讨论点**：是否强制使用模型的 Tool Calling 功能？如何编写一套既灵活又能严格约束输出结构的 System Prompt？

**当前结论：**
*   采用 structured output 是合理的，但前提是先由前端验证哪些组件和字段真的可渲染、可解析。
*   不建议让 LLM 直接输出真实前端组件 props，而应输出受约束的中间协议，如 `type + anchor + payload + priority`。
*   组件类型必须使用有限枚举，禁止模型自由发明新组件。
*   本项目的流程应从“先调 LLM 输出”调整为“前端先行验证组件与解析能力，再冻结 schema，最后再让 LLM 适配”。

### Q4：流式渲染与加载体验 (Streaming UX)
*   **问题**：V2 包含更多 UI 描述，Token 消耗可能增加。
*   **讨论点**：如何实现“边生成边显示”？如何避免页面在渲染过程中的频繁抖动？

**当前结论：**
*   第一阶段不把 streaming 作为主目标，先优先验证静态渲染和交互是否稳定。
*   更稳的策略是“正文与翻译层先出，标注层后挂载”，避免长文在小程序里频繁插入节点导致抖动。
*   如果后续要做流式渲染，建议按句块或段块增量注入，而不是按单条 annotation 实时注入。

### Q5：语法适配的深度 (Pedagogical Coverage)
*   **问题**：现有的 UI 组件能否完全覆盖英语语法的各种复杂情况（如：非限定性定语从句、倒装句、虚拟语气）？
*   **讨论点**：我们需要定义哪些高级视觉组件（如：指向箭头的 `RelationalArrow`）来辅助理解语法结构？

**当前结论：**
*   语法覆盖不应通过无限增加组件数量来解决，而应通过少量稳定渲染原语 + 更强的表达能力来承载。
*   第一阶段优先支持轻量语法标记和句级分析卡片，不优先支持复杂句法树、箭头连线等高成本视觉组件。
*   对复杂语法现象的支持重点放在“能表达清楚、能稳定渲染”，而不是一开始追求视觉最复杂的形式。

### Q6：原文中的标注定位问题 (Anchoring In Rendered Text)
*   **问题**：即使 AI 能选对组件，也不代表它返回的锚点一定能在小程序前端稳定命中原文。文本清洗、分句、换行、标点归一化后，原始锚点可能发生偏移或失配。
*   **讨论点**：V2 是否仍然坚持 `sentence_id` 作为硬边界？组件锚点是否应分层为 `word/phrase`、`sentence`、`paragraph`、`between_sentences` 几种类型？当前端实际渲染与模型理解不一致时，由谁负责做最终解析和降级？

**当前结论：**
*   定位责任不能完全留给前端，必须由后端先做一轮锚点解析、校验和降级。
*   前端可以在实验阶段验证 `sentence_id + 子串匹配` 是否易于实现，但正式接口应尽量返回已经过本地解析的稳定定位结果。
*   V2 需要把锚点类型显式分层：`word/phrase`、`sentence`、`paragraph`、`between_sentences`。
*   对不连续语法模式和分隔式短语，应统一采用 span 列表 / 锚点列表协议，而不是继续沿用单一 span 假设。

### Q7：翻译呈现方式与双语阅读模式 (Bilingual Reading)
*   **问题**：产品更倾向于双语模式阅读，即“一行英文原文，下面显示中文”。这意味着翻译层可能不是 AI 自由选择的标注组件，而是结果页的固定渲染层。
*   **讨论点**：翻译是否应从“标注系统”中拆出，作为页面级固定能力处理？双语模式、沉浸模式、精读模式之间如何切换？逐句翻译、段落翻译和更细粒度的对照关系应如何与其他标注共存而不造成版面拥挤？

**当前结论：**
*   翻译层从“自由标注组件”中拆出，作为结果页固定能力处理。
*   页面结构建议固定为：英文主层 + 中文翻译副层 + 标注叠加层。
*   第一阶段优先支持“英文一行、中文一行”的逐句双语模式，不做逐词行间对照。
*   沉浸模式、双语模式、精读模式应由页面配置和前端控制条切换，而不是交由 AI 决定。

### Q8：词典点击与 AI 标注点击的交互冲突 (Dictionary vs Annotation Interaction)
*   **问题**：前端会调用词典 API 做实时单词查询，用户点击单词即可查看释义。如果 AI 标注也依赖点击交互，就会与词典查询争抢同一个入口，导致手势语义冲突。
*   **讨论点**：是否应把“点击单词查词”固定为系统级基础交互，而让 AI 标注更多通过颜色、角标、边注、句尾入口、底部面板等方式触发？词典释义与 AI 讲解是否需要合并到同一个弹层中分区展示，还是必须严格分离？

**当前结论：**
*   “点击单词查词”固定为系统级基础交互，优先级高于 AI 标注点击。
*   词级 AI 标注可以借用同一个点击入口，但应统一进入同一个词级弹层，在其中分区展示“词典结果”和“AI 补充解释”。
*   语法和句级标注不应与单词点击争抢入口，应优先通过句尾入口、边注、段间卡片或底部面板承载。
*   这意味着词级组件与句级/语法级组件在交互层级上必须做明确分工，而不是都依赖点击正文触发。

## 5. 当前协作流程调整

基于以上 8 个问题的讨论结论，V2 的推进顺序做如下调整：

1. **先讨论清楚约束与边界**：优先锁定锚点协议、双语模式、点击优先级、冲突规则。
2. **前端先行**：前端先开发标注结果页实验场，验证在微信小程序环境下哪些渲染组件和交互方式真正稳定可行。
3. **再冻结 Schema**：基于前端验证结果，反推出一套可稳定解析的结构化数据格式。
4. **最后适配 LLM 输出**：等结构化协议稳定后，再让 LLM 针对该 schema 做 structured output 测试与联调优化。
5. **尽早打通联调路径**：最好让后端接口尽快接到前端结果页，形成真实的本地联调链路，而不是长期停留在 mock 数据阶段。
