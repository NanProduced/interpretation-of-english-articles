# 微信小程序技术边界

来源文档：`微信小程序技术边界与项目架构调研`

本文档只提炼直接影响开发分工和架构判断的部分，用来回答一个核心问题：哪些能力可以依赖微信小程序或微信生态，哪些必须由我们自己实现。

## 基本判断

微信小程序本质上是运行在微信客户端中的前端应用，不是后端运行环境。

因此本项目必须采用前后端分离结构：

- 小程序前端负责页面、交互、渲染、请求发起
- 后端负责 Workflow、模型调用、业务 API 和服务端安全逻辑

## 小程序原生或微信生态可以提供的能力

以下能力可以直接用小程序原生能力或微信生态支持完成：

### 前端界面与交互

- 页面渲染
- 列表、弹层、滚动区域
- 自定义组件
- 本地状态管理
- 剪贴板读写
- Canvas 绘制

### 平台能力

- 微信登录
- 本地存储
- 分享能力
- 微信开发者工具调试

### 微信云能力

- 云数据库
- 云函数
- 云托管

### 可后续考虑的原生能力

- OCR 相关能力
- 订阅消息

## 必须由我们自己实现的能力

以下能力不能指望小程序原生直接完成，必须由我们自己设计和实现：

### 1. AI Workflow

多步解读链路、Agent 编排、结构化输出校验都必须放在后端。

### 2. 模型调用

大模型 API Key 不能放在前端，所有模型调用必须由后端完成。

### 3. 结构化输出契约

小程序不提供“结构化解读协议”，输出 Schema、校验规则、版本控制都必须由我们自己定义。

### 4. 业务层逻辑

例如：

- profile 映射
- 词汇优先级规则
- 失败重试
- 降级策略
- 缓存策略

这些都是我们自己的业务实现，不是微信提供的能力。

### 5. 结果页核心体验

小程序只能提供基础渲染能力，真正决定产品差异化的标注渲染、分层展开、长文性能优化都要我们自己设计。

## 推荐架构边界

当前建议采用混合架构：

### 小程序前端

负责：

- 文章输入
- 用户配置
- 结果页渲染
- 用户交互

推荐技术：

- Taro + React + TypeScript

### 云函数或轻服务层

可负责：

- 内容安全审核调用
- 简单 CRUD
- 定时任务

这层适合轻逻辑，不应承担核心 AI Workflow。

### 云托管后端

负责：

- FastAPI 服务
- LangGraph 编排
- PydanticAI Agent
- 结构化输出
- 模型调用

这是项目的核心后端。

## 当前最重要的边界判断

### 哪些可以后置

- OCR
- 复杂运营能力
- 学习数据统计
- 高级推荐系统

### 哪些不能后置

- 内容安全审核
- 前后端职责边界
- 输出 Schema
- 结果页渲染模型
- 后端虚拟环境和依赖管理规范

## 对当前开发的直接指导

当前阶段应坚持：

- 前端先只做页面与交互，不做模型调用
- 后端先把最小 Workflow 跑通
- 小程序能力只作为“承载与展示”，不要误把平台能力当作产品能力
- 所有真正影响结果质量的逻辑，都要沉到后端和规则层
