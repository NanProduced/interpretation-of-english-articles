# 微信小程序产品与技术边界

来源文档：

- `PRD · Claread透读`
- 微信小程序官方文档

本文档用于回答两个问题：

1. 从 Claread透读 的产品目标出发，小程序端到底应该具备哪些能力
2. 这些能力中，哪些适合放在小程序前端，哪些必须放在后端，哪些可以借助微信平台能力完成

## 一、基本判断

Claread透读不是一个“把 LLM 结果塞进页面”的 demo，而是一个面向非零基础英语用户的阅读工具型产品。

根据 [产品需求摘要](../product/prd-summary.md)，它的核心价值是：

- 不是普通翻译，而是“教你读懂”
- 不是通用聊天，而是围绕英文阅读场景的专用产品
- 通过稳定的结构化输出和精美可交互的结果页，提供明显优于通用 AI 对话的体验

因此，微信小程序在这个项目中的角色应被定义为：

- 用户输入入口
- 用户状态与页面状态承载层
- 结果页渲染与交互层
- 微信平台能力接入层

而不是：

- AI 能力本身
- 核心业务判断层
- 结构化输出真相来源

一句话说：

- 小程序负责“把产品体验做出来”
- 后端负责“把 AI 能力做对、做稳、做可控”

## 二、从 PRD 反推的小程序核心能力

结合 PRD，当前阶段小程序应该具备以下核心能力。

### 1. 文章输入能力

这是主入口，必须是产品级能力。

至少应支持：

- 文本粘贴
- 手动输入与编辑
- 清空与重新提交
- 输入校验
- 提交前安全检查流程

PRD 已明确：MVP 以“用户自己带文章来解读”为主，不是以每日推荐文章为主。

### 2. 用户配置能力

配置应围绕“使用目的”，而不是技术参数或自评水平。

当前主配置路径应围绕：

- 日常阅读提升
- 考试备考
- 学术/专业阅读

这里的小程序职责是收集“用户用途”，不是暴露复杂 profile 细节。真正的内部 profile 映射应由后端完成。

### 3. AI 解读发起能力

小程序需要有完整的分析发起与状态反馈能力，包括：

- 提交分析
- 分析中状态
- 超时与失败提示
- 重试
- 回到输入页修改后重提

这不是单纯一个按钮，而是一条完整产品链路。

### 4. 结果页能力

这是 Claread 的核心差异化能力，小程序必须承担这层体验。

至少应支持：

- 原文分段展示
- 重点词汇高亮
- 行内点击查看
- 深度层展开
- 语法与长难句查看
- 逐句翻译
- warning / 降级结果展示
- 空结果与失败结果展示

### 5. 分层阅读能力

根据 PRD，结果页应采用渐进式展开：

- 默认层：原文 + 重点高亮
- 快查层：点击词汇或短语查看快速解释
- 深度层：语法、句解、翻译
- 篇章层：后续逐步开放

这意味着结果页不能做成“所有信息一次性平铺”。

### 6. 最小用户资产能力

如果没有这层，产品会退化成一次性分析工具。

最小闭环建议包括：

- 最近记录
- 结果回看
- 再次分析
- 收藏或置顶
- 分析失败后的恢复入口

### 7. 首页与轻引导能力

PRD 明确说“每日文章不是核心，但可以降低首次使用门槛”。

所以首页应有：

- 直接开始输入文章的主入口
- 少量推荐文章或示例文章
- 最近使用记录入口

但不能让首页运营内容压过“输入文章开始解读”这个主动作。

## 三、微信小程序平台能提供什么

微信小程序官方能力足够支持 Claread 做出完整产品壳，但这些能力主要是“承载与连接能力”，不是“AI 产品能力”。

### 1. 登录与身份入口

官方 `wx.login` 可以获取登录凭证 `code`，再由服务端换取 `openid / unionid / session_key`。

这意味着：

- 小程序可以承担轻量登录入口
- 真实用户身份建立必须放在后端

来源：[wx.login 官方文档](https://developers.weixin.qq.com/miniprogram/dev/api/open-api/login/wx.login.html)

### 2. 本地存储与缓存

官方 `wx.setStorageSync` 支持本地缓存，但有明确限制：

- 单个 key 最大 1MB
- 总存储上限 10MB
- 不适合作为运行时全局状态管理

因此：

- 可用于缓存最近输入、最近结果摘要、恢复态信息
- 不适合长期保存大量完整分析结果

来源：[wx.setStorageSync 官方文档](https://developers.weixin.qq.com/miniprogram/dev/api/storage/wx.setStorageSync.html)

### 3. 生命周期与退出恢复

页面支持：

- `onLoad`
- `onShow`
- `onHide`
- `onUnload`
- `onSaveExitState`

这非常适合处理：

- 分析中切后台
- 重回前台恢复状态
- 页面退出前保存恢复信息

来源：[Page 官方文档](https://developers.weixin.qq.com/miniprogram/dev/reference/api/Page.html)

### 4. 路由与页面间通信

官方 `wx.navigateTo` 支持页面跳转和 `EventChannel`，但：

- 不能跳 tabBar 页面
- 页面栈最多 10 层

因此小程序页面设计必须控制层级，不适合过深的多层流转。

来源：[wx.navigateTo 官方文档](https://developers.weixin.qq.com/miniprogram/dev/api/route/wx.navigateTo.html)

### 5. 分享能力

官方支持：

- 页面内分享
- 右上角转发
- 分享到朋友圈

但官方也明确强调：

- 分享应是用户自发行为
- 不应做诱导分享或强制分享解锁

来源：

- [转发官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/share.html)
- [Page.onShareAppMessage / onShareTimeline](https://developers.weixin.qq.com/miniprogram/dev/reference/api/Page.html)

### 6. 订阅消息

官方支持订阅消息，但它依赖模板、用户授权、服务端下发，更适合异步提醒型场景。

这说明：

- 订阅消息适合作为后续增强能力
- 不适合作为当前主路径刚需

来源：[订阅消息开发指南](https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/subscribe-message.html)

### 7. 启动性能与真机体验约束

官方对启动性能要求非常明确：

- 小程序启动体验会直接影响用户流失
- 首页应尽量轻
- 首屏渲染与代码包体积要重点关注

这对 Claread 的直接影响是：

- 首页不能过重
- 结果页长文渲染要克制
- 各种大数据结构不能无节制塞给前端

来源：[启动性能官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/performance/tips/start.html)

### 8. 安全要求

官方安全指引明确强调：

- 不信任用户数据
- 前端代码可被获取
- 重要业务逻辑放后台
- 后台接口必须鉴权

因此：

- 内容安全审核、AI 调用、用户权限、结果访问规则都不能放前端
- 前端只能做体验层校验，不能做最终安全判断

来源：[安全指引官方文档](https://developers.weixin.qq.com/miniprogram/dev/framework/security.html)

## 四、研究结论：云开发 / 云托管 / 数据库 / 未来 RAG

以下内容是当前阶段的研究结论，用于后续架构决策参考，不视为最终定案。

### 1. 云开发与云托管的角色更适合分层使用

- 云开发更像小程序配套基础设施，适合承接登录、数据库、存储、轻量云函数和环境集成
- 云托管更像后端运行层，适合承接 Claread 现有 Python / LangGraph 服务、AI workflow 和后续长任务
- 小程序直接调用云开发 AI 能力在技术上可行，但当前不适合作为主架构，因为会削弱后端对 prompt、schema、repair、审核与规则的统一控制

参考资料：

- [云开发小程序快速开始](https://docs.cloudbase.net/quick-start/mini-program/introduce)
- [云托管概述](https://docs.cloudbase.net/run/introduction)
- [小程序接入云开发 AI](https://docs.cloudbase.net/ai/quickstart/miniprogram)

### 2. 数据库更适合使用 CloudBase 托管能力，而不是在云托管容器里自建

- 从产品和运维角度看，云托管更适合跑应用服务，不适合承载数据库本体
- 当前业务数据更适合放在 CloudBase 托管数据库中，再由云托管后端统一访问
- 若考虑用户、分析任务、历史记录、收藏、反馈、权限等业务实体，MySQL 型数据库更像主业务库候选
- 文档型数据库也有价值，更适合放半结构化配置、调试记录或不值得严格建模的附属数据

参考资料：

- [云数据库概述](https://docs.cloudbase.net/en/database/intro/introduce-db)
- [云数据库（MySQL 型）访问](https://docs.cloudbase.net/database/access/tdsql)
- [云托管访问 CloudBase 云数据库](https://docs.cloudbase.net/run/best-practice/using-cloudbase-database)

### 3. 未来若接入 RAG，更适合与主业务库分层

- 当前研究更支持“业务数据库”和“检索数据库”分层，而不是把 RAG 直接塞进主业务库设计中
- 一种可行路线是：主业务数据继续走 CloudBase 数据库，RAG 由后端按需编排并调用独立检索层
- 检索层后续可优先评估 CloudBase 知识库能力；若能力不匹配，再接独立向量库或自建检索服务

参考资料：

- [知识库概述](https://docs.cloudbase.net/ai/agent/knowledgebase)
- [知识库检索 API](https://docs.cloudbase.net/http-api/knowledge/search)

### 4. 当前阶段更稳妥的候选方向

- 小程序前端：输入、配置、发起分析、结果页、恢复、分享、历史记录
- 云开发：登录、数据库、存储、轻量函数、环境接入
- 云托管：后端 API、AI workflow、结构化输出、规则中心、未来 RAG 编排

该方向的优势是：

- 前后端职责清晰
- 小程序不直连核心 AI 能力
- 数据与检索层后续都有扩展空间

## 五、能力边界：前端 / 平台 / 后端怎么分

下面按 Claread 当前阶段最重要的产品能力来拆。

| 能力 | 小程序前端 | 微信平台能力 | 后端 |
|---|---|---|---|
| 文章输入 | 页面、输入框、粘贴、清空、提交流程 | 剪贴板、页面交互 | 输入校验规则、内容安全审核、请求接收 |
| 用户配置 | 用途选择 UI | 本地缓存 | profile 映射、默认策略、规则中心 |
| AI 解读发起 | 请求发起、loading、失败重试 | 网络请求能力 | workflow、模型调用、结构化输出 |
| 结果页渲染 | 原文、标注、弹层、展开/收起、warning 展示 | 页面能力、滚动、组件系统 | 输出 contract、渲染协议、降级规则 |
| 历史记录 | 列表展示、回看入口 | 本地缓存可做轻量恢复 | 持久化记录、分页、删除、收藏 |
| 用户登录 | 登录触发、登录态 UI | `wx.login` | session 建立、用户识别、鉴权 |
| 分享 | 分享按钮、分享入口、结果页分享 UI | 转发、朋友圈分享 | 分享路径、分享态数据准备 |
| 状态恢复 | 前台/后台状态处理 | 生命周期能力 | 任务状态、记录状态、幂等恢复 |
| 数据存储 | 本地轻缓存与恢复态 | 本地存储能力 | 主业务数据库、对象存储、数据访问控制 |
| 未来 RAG | 不直接承载检索逻辑 | 无核心业务能力 | 检索层选择、召回编排、知识库接入 |
| 内容安全 | 触发提交流程、展示拦截结果 | 无直接业务能力 | 审核逻辑、审核结果判定 |
| 埋点与异常 | 页面埋点、前端异常上报 | 小程序运行环境 | 服务端日志、分析链路观测 |

## 六、哪些必须由我们自己实现

以下能力不能指望微信小程序原生直接完成，必须由我们自己设计实现。

### 1. AI Workflow

包括：

- 多 agent 编排
- prompt 策略
- 输出稳定化
- repair 与降级

这一整层必须在后端。

### 2. 模型调用

API Key 和模型路由不能放前端，必须由后端统一管理。

### 3. 结构化输出契约

小程序没有“英文文章解读协议”这种能力。

所以：

- schema
- 校验规则
- projection 逻辑
- 版本控制

都必须由我们自己定义。

### 4. 业务规则与安全控制

包括：

- profile 映射
- 词汇优先级规则
- 失败重试
- 降级策略
- 接口鉴权
- 内容安全

这些都必须以后端为准。

### 5. 结果页差异化体验

小程序只提供基础渲染能力。

真正决定 Claread 产品价值的仍然是我们自己设计的：

- 标注呈现方式
- 分层阅读体验
- 长文渲染策略
- 交互细节

## 七、当前阶段的产品能力判断

### 1. 当前阶段必须具备

这是 Claread 当前阶段不做就不成立的能力。

- 文本输入与提交
- 用途配置
- 真实 `/analyze` 联调
- 结果页分层渲染
- loading / error / empty / degraded 状态
- 最近记录与回看
- 基础登录态
- 本地恢复与缓存
- 分享入口
- 埋点与异常上报

### 2. 当前阶段应该尽快具备

- 收藏与置顶
- 重新分析历史文本
- 用户反馈入口
- 弱网与超时恢复
- 真机性能优化
- 页面退出恢复与后台恢复

### 3. 当前阶段可以后置

- OCR 导入
- 订阅消息提醒
- 学习统计
- 高级配置中心
- PDF 导出
- 篇章级更复杂分析展示
- 高级推荐系统

## 八、对当前开发的直接指导

当前阶段应坚持以下原则。

### 1. 小程序先做“产品壳”，不是做“AI 本体”

前端优先建设：

- 真实输入链路
- 结果页完整状态
- 用户回看闭环
- 微信平台能力接入

### 2. 后端继续承担“AI 真相层”

后端优先建设：

- workflow 质量
- 输出稳定性
- 结构化协议
- 安全与鉴权

### 3. 小程序不要假设后端永远返回理想结果

页面必须接得住：

- 无标注但有翻译
- 有 warning
- 局部降级
- 超时失败
- 重新分析

### 4. 不要把平台能力误认为产品能力

微信提供的是：

- 登录入口
- 分享入口
- 生命周期
- 缓存
- 页面容器

但 Claread 真正的产品能力，仍然来自：

- 后端 workflow
- 输出 contract
- 结果页交互设计

## 九、最终结论

从 PRD 和微信官方能力边界综合来看，Claread 小程序当前应被定义为：

- 一个低摩擦的英文阅读工作台
- 一个承载 AI 阅读体验的产品壳
- 一个真正适配微信小程序环境的内容输入、结果查看与回看工具

而不是：

- 一个前端直接调用模型的 AI 页面
- 一个只展示静态结果的 demo
- 一个只靠微信平台能力就能成立的产品

最重要的判断是：

- 小程序必须完整承担“输入、状态、渲染、交互、恢复、分享、用户路径”
- 后端必须完整承担“分析、结构化、安全、规则、存储真相”

只有这两层都建立起来，Claread 才能从“后端可跑”真正走向“产品可用”。
